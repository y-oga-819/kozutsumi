# 機能仕様

UI 機能の詳細と成功指標をまとめる。アーキテクチャ上の機構（[architecture.md](./architecture.md)）を、どんな画面・操作で見せるか。

## 1. 機能設計

### 1.1 デイタイムライン

1 日の時間構造を横バーで可視化。

- 9:00〜18:00 を基本、予定がある場合は動的に伸びる
- イベントはプロジェクト色のブロック、各ブロックに所要時間
- 空き時間ブロックに所要時間
- 現在時刻の緑マーカー、現在の空き時間をハイライト

### 1.2 イベントカード一覧

タイムラインバーの下にイベントを時系列のカードで表示。

- 時刻、所要時間、タイトル
- 添付資料ありの場合はドキュメントアイコン
- 会議 URL ありの場合はビデオアイコン
- NEXT イベントには「Meet に参加」ボタンを直接配置
- カードタップで詳細パネル

### 1.3 タスクスタック

スタックの最上位を大きなカードで表示し、「今これをやる」を明確にする。

- **最上位**: プロジェクト色、タイトル、補正後見積もり時間、依存イベント、完了条件（Goal / Done / First step）
- **2 番目以降**: コンパクトな 1 行
- ドラッグ&ドロップで並べ替え（モバイルタッチ対応）

ユーザーが触る動詞は **start / stop / complete + 1-tap 割り込み** だけ（[ADR-0058](../adr/0058-timer-three-verbs-and-no-ai-interruption.md)）:

- **start ボタン**: トップタスクを active 化、タイマー表示開始
- **stop ボタン**: active → paused（理由は問わない、1 タップ主義）
- **complete ボタン**: 実作業時間を自動記録して done へ移動 + スタックの 2 番目が top に昇格
- **⚡ 割り込みボタン**: 1 タップで現在タスクを中断 + 割り込みイベント記録（事前分類なし、[ADR-0059](../adr/0059-one-tap-interrupt-recording.md)）

「重いから後でやる」は stop で paused にすれば事実上の push back と同等。stack 並び替えで別タスクを top にして再 start する。warmup / モーダル / 選択肢は一切出さない。

カードタップで Markdown 詳細パネル + 完了条件編集（AI が後追いで補完した内容を確認・修正できる、[ADR-0064](../adr/0064-task-creation-title-only-with-ai-template-fill.md)）。

### 1.4 朝の棚卸し画面

1 日の最初に開く専用画面（**default opt-in**、自動起動しない、[ADR-0062](../adr/0062-morning-review-ritual-fifteen-minutes.md)）。

- **昨日の summary**: 完了 / 中断 / 割り込み件数 + 行動データの示唆（Whoop Recovery Score 風ゾーン表示、「あなたの平均」との相対比較）
- **今日の stack 確認 + 並び替え**: 重い順 / 軽い順の自動並び替えボタン + 手動 drag&drop
- **「今日の意図」選択**: 1 つだけ宣言
- **過集中の事後補正**: 長時間記録された timer タスクの実作業時間補正 + 「時間に見合う成果が出たか」の妥当性振り返り
- 15 分で完結するキャップ

説教調を避ける文言設計（Whoop 型: 数値ゾーンに翻訳し、判断は user に委ねる）。

### 1.5 Tree View（履歴ビュー）

git log tree 風の UI で過去の活動を振り返る。

- 縦軸が時間、横軸がプロジェクト
- 各プロジェクトが色付きの縦線（ブランチ）
- 完了タスクがノード、割り込みタスクは別マーク
- 日付区切り

### 1.6 Wrapped（蓄積後インサイト面）

差別化軸の **深層** を見せる場（[ADR-0057](../adr/0057-redefine-moat-as-goal-driven-ai-decomposition.md)）。週次 / 四半期 / 年次のナラティブを AI が生成。

- 「今週最も集中した時間帯」「最もエネルギーを注いだプロジェクト」「先送りされたタスク」
- 「今年最も集中した時間帯」「成長した分野」「苦手分野の変化」
- 説教調を避ける文言設計（Whoop / Spotify Wrapped 型）

3 ヶ月後・半年後・1 年後にしか出せない深さで「秘書らしさ」を見せる。


## 2. 成功指標（KPI）

**Goodhart's Law 回避の原則**: 単一指標を絶対基準で目標化しない。タスクサイズ別カウント、「あなたの平均」との相対比較、定性的観察を併用する。「見積もり精度 ±20%」のような per-task 計測 KPI は中核から外す（[ADR-0058](../adr/0058-timer-three-verbs-and-no-ai-interruption.md) / architecture §1.8 参照）。

### 2.1 日次

- **タスクサイズ別完了数**（S / M / L 別）: LeetCode 型。「完了数 = 価値」の誤算を回避
- **棚卸し開封率**: 起動日のうち朝の棚卸しを開いた割合。定着の早期シグナル
- **スタック消化率**（参考）: 完了割合。目標値は固定しない

### 2.2 週次

- **週次 Wrapped 開封率**: 定着の核指標
- **プロジェクトバランス**: 本業 vs それ以外の消化時間比率（参考）
- **割り込み率**: 全タスクに対する割り込み発生の割合
- **回避タスク検出数**: 3 日以上連続で後回しにされたタスクの数

### 2.3 継続

- **連続利用日数**: cumulative（戻ってきた日を肯定する。streak 型ではない）
- **棚卸し利用パターン**: 毎朝 / 週末まとめて / 散発的 等の質的観察
- **インサイト精度**: Wrapped で出した「あなたは○○な傾向」に対する user の合意率（定性的観察、四半期に 1 回振り返る）
- **放棄パターン**: 使わなくなった日の前後の状況分析
