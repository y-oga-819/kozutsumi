# アーキテクチャ

kozutsumi をどう作るかを示す。解決策の構造（メカニズム）、データモデル、技術スタックをまとめる。

## 1. 解決策の構造

### 1.1 2層の価値モデル

**表層（即時的価値）:** タスク管理ツールとしての使いやすさ。イベント駆動スタック、タイムラインバー、割り込みのpush/pop、git tree履歴。AIなしでも成立する。これが継続利用の理由になる。

**深層（蓄積的価値）:** 使い続けることで「自分専用のAI秘書」が育つ。3ヶ月後には行動パターンに基づくパーソナライズが効き始める。ここが競合に到達できない領域。

コールドスタート問題はこの2層構造で解く。Phase 1-2では表層の価値だけで使い続ける理由がある。AIの精度が低い期間もツールとして成立する。フィードバックは自然に蓄積され、Phase 3-4でAIが効き始めたとき、「普通に便利だから使ってたら、いつの間にかAIが賢くなってた」という体験になる。

### 1.2 イベント駆動スタック

イベントをカレンダーから取得しタイムライン上に固定する。イベント間の「空き時間」を可視化し、タスクはその空き時間の中でスタックから1つずつ消化する。これにより時間固定のイベントとタスクが衝突する問題を構造的に解消する（スタック型タスク管理の挫折要因の詳細は [vision.md](./vision.md) 参照）。

### 1.3 優先度主導のスタック

スタックの順序は純粋に優先度で決まる。空き時間へのフィッティングは副次的。

優先度の高い45分タスクがスタックのトップにあり、今の空きが30分なら、2つのアプローチがある。

**分断着手:** 30分だけ進めてpausedにし、MTG後に残り15分をやる。後続タスクのボトルネックになっている場合はこちらが正しい。

**分解挿入:** 「構成のアウトラインだけ作る（10min）」「本文を書く（20min）」「レビュー・修正（15min）」に分解し、10minのタスクを今の空きで消化する。全体像が見えず着手ハードルが高い場合はこちらが正しい。

AIは過去の行動パターンからどちらが適切かを判断する。「このユーザーはドキュメント系タスクで最初のactive区間が短い（着手して5分で中断するパターンが多い）→ 構成で迷っている可能性が高い → アウトラインタスクを先に挿入する」。

### 1.4 割り込みのスタック表現

割り込みが発生したとき、スタックのpush/popで構造的に対応する。

1. 現在のタスクが自動でpaused
2. 割り込みタスクがスタックのトップにpush（⚡マーク付き）
3. 割り込み完了で元のタスクがトップに戻る

割り込みの記録は「本来やりたかったことができなかった理由」の分析データになる。「今週は割り込みが12件、予定していた作業の30%が押し出された」「火曜の午後に割り込みが集中する傾向がある」。

### 1.5 見積もり補正エンジン

ユーザーの見積もりを過去実績データで自動補正する。

**タスク種類別の補正倍率:**
- コーディング系: 見積もりの0.8倍で終わる（得意）
- ドキュメント系: 見積もりの2.2倍かかる（苦手）
- 調査系: ばらつきが大きい（0.5〜3倍）

**時間帯×タスク種類のクロス分析:**
- 月曜午前のコーディング: 見積もり通り
- 金曜午後のドキュメント: 3倍かかる

**補正の見せ方:**
「あなたの見積もりは信用できません」ではなく「過去の実績から、このタスクには45min確保しています」とさりげなく補正済みの数字を見せる。スタック構成やスケジューリングには補正後の数字が使われるが、ユーザーの入力は尊重する。補正が的確であれば信頼が積み上がる。

### 1.6 行動パターン分析

見積もり補正の上位概念として、タスク消化全体の行動パターンを分析する。

**着手パターン:** タスクの種類ごとに、着手までの時間・初回active区間の長さを記録。初回activeが短いタスク種類は「着手で迷う」傾向→アウトラインや調査の前段タスクを自動挿入。

**回避パターン:** 3日連続でスタックから除外されたタスクを検出。「SLI定義書の更新を3日連続で後回しにしています。金曜のレビューMTGまでに完了が必要です。今日の午前に配置しますか？」

**エネルギー変動:** 時間帯別のタスク完了速度を記録。「午前は集中力が高い→重いタスクを午前に配置」「MTG直後は勢いがある→避けがちなドキュメント系をMTG直後に配置」。

**苦手分析:** 見積もり超過率が高いタスク種類について、行動ログ（active/paused遷移の頻度、各active区間の長さ）から原因を推定。
- active→pausedが頻繁 → 集中できていない、または方向性に迷っている
- 最初のactiveが長いが合計時間が超過 → 完成度の基準が高すぎる
- 着手自体が遅い → 心理的ハードルが高い

AIはパターンに応じた対処をタスク構造で提案する。着手の迷い→前段タスク挿入、完成度の問題→完了条件の明確化、心理的ハードル→タスクの細分化。

### 1.7 暗黙的フィードバック

ユーザーに「フィードバックしてください」と明示的に求めない。自然な操作が全てフィードバックになる。

- タスクをそのまま消化 → 提案の承認
- 並べ替え → 優先度判断の修正データ
- タスクを削除 → 不要なタスクの学習
- タスクのタイトルを書き換え → 分解精度の修正データ
- 2つのタスクを統合 → 分解粒度が細かすぎた
- さらに分割 → 分解粒度が粗すぎた
- 完了までの実時間 → 見積もり精度データ
- active/paused遷移のタイミング → 行動パターンデータ
- 割り込みの発生タイミング → 割り込みパターンデータ

タスク分解も同様。AIが分解した結果はそのままスタックに挿入される。承認ステップは挟まない。ユーザーが削除・書き換え・統合・再分割した操作が暗黙的にフィードバックされ、分解の精度が上がっていく。

### 1.8 スコアリング × LLM の二段階提案

直近1週間のイベントとタスクの全体像から「今日のスタック」を提案する。

**第1段階: スコアリングモデル（数値最適化）**

```
score = w1 × (1 / 依存イベントまでの実質残り日数)
      + w2 × ボトルネック度（後続タスクの依存数）
      + w3 × プロジェクトバランス補正
      + w4 × 本業タスク優先バイアス
      + w5 × 曜日パターン係数
      + w6 × 連続回避ペナルティ
      + w7 × エネルギー適合度（時間帯×タスク種類）
```

「実質残り日数」は単純な日数ではなく、明日以降の空き時間を考慮した実質的な作業可能日数。「依存イベントまで3日あるが明日明後日は予定ぎっしり→実質残り1日」。

**第2段階: LLM妥当性検証（文脈判断）**

スコアリングの出力に対して、LLMが文脈的チェックを行う。

- 面接が2つ重なる日に60minタスクを2つ積むのは非現実的
- このMTGの事前資料作成がスタックに含まれていない
- 今週の本業タスク比率が低すぎる
- ドキュメント系タスクが午後に配置されているが、このユーザーは午前のほうが効率がいい
- 3日連続で回避されているタスクに介入すべき

### 1.9 2つのビュー

**Stack View（未来）** — 「次に何をやるか」を1つだけ見せる。

**Tree View（過去）** — git log treeのメタファーで過去の活動を振り返る。本質的にはWBSの別UI。


## 2. 情報構造

### 2.1 階層モデル

```
main（自分）
├── project: 転職活動
│   ├── epic: Dirbato選考
│   │   ├── story: 最終面接準備
│   │   │   ├── task: 志望動機パターンA作成 (15min)
│   │   │   ├── task: 企業研究メモ更新 (20min)
│   │   │   └── task: 逆質問3つ整理 (10min)
│   │   └── story: オファー条件交渉
│   └── epic: ULS検討
├── project: SLO推進
└── project: 負荷試験環境構築
```

実運用はproject + taskの2階層を基本とし、epic/storyはオプショナルな中間層。

### 2.2 タスク

- 見積もり時間: 分単位（ユーザー入力）
- 補正後見積もり時間: 過去実績から自動算出
- 依存イベント: カレンダーイベントへのリンク（任意）
- プロジェクト: 所属プロジェクト
- 詳細: Markdown
- 割り込みフラグ: 割り込みで挿入されたかどうか
- 分解元タスク: AI分解で生まれた場合、親タスクへのリンク

### 2.3 タスクのステート

```
idle → active → paused → active → done
```

各遷移で記録するコンテキスト:
- 遷移時刻
- paused理由（MTG開始/割り込み/自発的中断）
- 直前のイベント（MTG直後かどうか）
- 時間帯
- 曜日

active区間の合計 = 実作業時間。この実作業時間と見積もりの乖離が、見積もり補正エンジンの入力データになる。

### 2.4 行動ログ

タスクのステートマシンとは別に、以下の行動データを記録する。これが kozutsumi の真のデータ資産。

- スタック内での位置変更（並べ替え）
- スタックからの除外（削除）
- タスクタイトルの書き換え
- タスクの統合・再分割
- 分解されたタスクの削除・修正
- 提案されたスタック順序 vs 実際の消化順序
- 割り込みの発生タイミングと種類

### 2.5 イベント

カレンダーから取得する時間固定の予定。

- 開始時刻・終了時刻
- タイトル
- 会議URL（Meet/Zoom、自動判定）
- 添付資料の有無
- 関連プロジェクト（任意）
- 説明文（Markdown）


## 3. 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| フロントエンド | Next.js | Webアプリとしてスマホ・デスクトップ両対応 |
| 認証 | Google OAuth | カレンダーAPI連携のOAuthトークン共用 |
| カレンダー連携 | Google Calendar API v3 | OAuth共用で企業Workspaceも取得可能 |
| AIサジェスト | Gemini 2.5 Flash | 無料枠で十分（10RPM、250リクエスト/日） |
| バックエンド/DB | Supabase (Postgres + Auth) | 個人ツールとしてAuth + DBで十分 |
