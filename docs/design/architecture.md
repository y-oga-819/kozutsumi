# アーキテクチャ

kozutsumi をどう作るかを示す。解決策の構造（メカニズム）、データモデル、技術スタックをまとめる。

## 1. 解決策の構造

### 1.1 2層の価値モデル

**表層（即時的価値）:** タスク管理ツールとしての使いやすさ。3 動詞 UX、ゴール駆動の AI 分解、イベント駆動スタック、タイムラインバー、git tree 履歴。AI なしでも成立する。これが継続利用の理由になる。

**深層（蓄積的価値）:** 使い続けることで「自分専用の AI 秘書」が育つ。3 ヶ月後・半年後・1 年後にしか出せないインサイト（週次 / 四半期 / 年次 Wrapped、朝の Recovery Score）が、機能比較に巻き込まれない独自軸になる。

コールドスタート問題はこの 2 層構造で解く。表層の価値だけで使い続ける理由がある期間に、フィードバックは自然に蓄積され、深層の AI 精度が効き始めたとき、「普通に便利だから使ってたら、いつの間にか AI が賢くなってた」という体験になる。

詳細は [vision.md](./vision.md) と [ADR-0057](../adr/0057-redefine-moat-as-goal-driven-ai-decomposition.md) 参照。

### 1.2 timer UX 原則 — 3 動詞だけ

ユーザーが触る動詞は **start / stop / complete の 3 つだけ**（[ADR-0058](../adr/0058-timer-three-verbs-and-no-ai-interruption.md)）。

- **stack top が timer の current task として auto-bind** される。「次に何をやるか」を選ぶ操作はゼロ
- **timer 動作中 (start ~ stop) には AI が能動的に介入しない**。warmup フェーズ・モーダル・選択肢を一切出さない
- AI による分解 / テンプレ補完 / 提案は **timer 文脈外**（タスク作成時 / タスク詳細画面 / 朝の棚卸し）でのみ発動

「重いから後でやる」は stop で paused にすれば事実上の push back と同等。stack 並び替えで別タスクを top にして再 start する。

### 1.3 イベント駆動スタック

イベントをカレンダーから取得しタイムライン上に固定する。イベント間の「空き時間」を可視化し、タスクはその空き時間の中でスタックから 1 つずつ消化する。これにより時間固定のイベントとタスクが衝突する問題を構造的に解消する（スタック型タスク管理の挫折要因の詳細は [vision.md](./vision.md) 参照）。

### 1.4 Calendar event の非対称 stop

scheduled event（Google Calendar 等から取得した予定）と timer の関係は **非対称**（[ADR-0060](../adr/0060-calendar-event-asymmetric-stop.md)）:

- **予定開始時刻に timer を自動停止**
- **予定終了時には自動再開しない（manual resume 必須）**

Externalize time（Barkley）と Autonomy 保持（SDT）の両立。Motion 型の自動スケジューリング（タスクを次々自動再開）を構造的に避ける。

### 1.5 1-tap 割り込み記録

突発割り込み（Slack / Notion / huddle 等）は専用ボタン **1 タップ** で（[ADR-0059](../adr/0059-one-tap-interrupt-recording.md)）:

1. 現在の timer を停止（paused）
2. 割り込みイベントを記録（時刻のみ、事前分類なし）

割り込みタスクは stack に push しない（旧設計の push/pop モデルから本軌道修正で離れた）。分類 / 振り返り / タスク化は朝の棚卸しで後追いする。

### 1.6 ゴール駆動の AI 分解

差別化の核（表層）。でかいタスクを **1 時間単位でゴールを追いかけられる粒度** に AI が分割する（[ADR-0061](../adr/0061-ai-decomposition-one-hour-target-and-done-condition-schema.md)）:

- 目標粒度: 30〜90 分レンジを推奨目安（強制ではない）
- 各子タスクに **完了条件（Goal / Done / First step）** を AI が言語化
- 親タスクの達成度を子の積み上げで可視化（ParallelogramProgress 系の延長、[ADR-0055](../adr/0055-parallelogram-progress-wrap-for-large-n.md)）
- AI 分解の発動は **timer 文脈外**: タスク作成時 / タスク詳細画面 / 朝の棚卸し

これが「ゴールが見えないタスクで詰まる」想定ユーザー特性への直接処方になる。

### 1.7 タスク作成 — title 必須のみ + AI 後追い補完

「タスク管理が目的になると面倒で続かない」を構造的に避ける（[ADR-0064](../adr/0064-task-creation-title-only-with-ai-template-fill.md)）:

- 必須入力は **title のみ**
- メモ書き（`body`）は推奨
- 残りの schema（Goal / Done / First step / Risk 等）は AI が後追いで補完

補完タイミングは **timer 文脈外**（タスク詳細画面 / 朝の棚卸し）。借金玉「自分を変えるな、道具に頼れ」/ Davis "morally neutral" の AI 自動化版。

### 1.8 見積もり補正エンジン

ユーザーの見積もりを過去実績データで自動補正する（[ADR-0024](../adr/0024-estimation-correction-by-category-median.md)〜[0026](../adr/0026-estimation-correction-display-style.md) 系）。

**タスク種類別の補正倍率の例:**
- コーディング系: 見積もりの 0.8 倍で終わる（得意）
- ドキュメント系: 見積もりの 2.2 倍かかる（苦手）
- 調査系: ばらつきが大きい（0.5〜3 倍）

**時間帯 × タスク種類のクロス分析:**
- 月曜午前のコーディング: 見積もり通り
- 金曜午後のドキュメント: 3 倍かかる

**補正の見せ方:**
「あなたの見積もりは信用できません」ではなく「過去の実績から、このタスクには 45min 確保しています」とさりげなく補正済みの数字を見せる。ユーザーを変えるのではなく道具で吸収する。

**Goodhart リスクへの留意:** 「見積もり精度 ±20% 以内」のような KPI を測定 = 目標化すると、per-task 計測自体が摩擦になる（MyFitnessPal 型「ダイエットコーラ +0 を律儀記録」の轍を踏む）。本機能の主目的は「見積もりを当てる」ではなく **「蓄積データから AI 分解の精度を上げる学習素材」** とする。

### 1.9 行動パターン分析

見積もり補正の上位概念として、タスク消化全体の行動パターンを分析する。

**着手パターン:** タスクの種類ごとに、着手までの時間・初回 active 区間の長さを記録。

**回避パターン:** 3 日連続でスタックから除外されたタスクを検出。朝の棚卸し（§1.12）で示唆。

**エネルギー変動:** 時間帯別のタスク完了速度を記録。Whoop 風 Recovery Score の材料。

**苦手分析:** 見積もり超過率が高いタスク種類について、行動ログ（active/paused 遷移の頻度、各 active 区間の長さ）から原因を推定。

これらの分析結果は **timer 動作中は出さない**。朝の棚卸し（§1.12）と Wrapped（§1.13）でユーザーに渡る。

### 1.10 暗黙的フィードバック

ユーザーに「フィードバックしてください」と明示的に求めない。自然な操作が全てフィードバックになる。

- タスクをそのまま消化 → 提案の承認
- 並べ替え → 優先度判断の修正データ
- タスクを削除 → 不要なタスクの学習
- タスクのタイトル / 完了条件を書き換え → 分解精度の修正データ
- 2 つのタスクを統合 → 分解粒度が細かすぎた
- さらに分割 → 分解粒度が粗すぎた
- 完了までの実時間 → 見積もり精度データ
- active/paused 遷移のタイミング → 行動パターンデータ
- 割り込みの発生タイミング → 割り込みパターンデータ
- 朝の棚卸しでの事後補正値 → 過集中 / 時間 vs 成果のシグナル

タスク分解も同様。AI が分解した結果はそのままスタックに挿入される。承認ステップは挟まない（[ADR-0017](../adr/0017-ai-task-decomposition-async.md)）。ユーザーが削除・書き換え・統合・再分割した操作が暗黙的にフィードバックされ、分解の精度が上がっていく。

### 1.11 過集中は介入しない

タイマー駆動の active session が長時間続いても、AI 側から能動的に中断 / 警告しない（[ADR-0063](../adr/0063-no-intervention-on-hyperfocus.md)）:

- 半熟ポモドーロ的な「乗っていても止める」soft nudge は採用しない
- 非アクティブ検知による auto-pause も default では入れない（opt-in 余地は残す）
- 強制中断（Heart 系 / Streak 系 / 強制 Pomodoro）は禁止

過集中で stop し忘れた場合は朝の棚卸し（§1.12）で **事後補正** する。Hallowell VAST「right kind of difficult」と整合。

### 1.12 朝 15 分の棚卸し儀式

1 日の最初に開く専用画面（[ADR-0062](../adr/0062-morning-review-ritual-fifteen-minutes.md)）:

- **昨日の summary**（完了 / 中断 / 割り込み件数 + 行動データの示唆、Whoop Recovery Score 風）
- **今日の stack 確認**（重い順 / 軽い順並び替え）
- **「今日の意図」を 1 つだけ宣言**
- **過集中の事後補正**（実作業時間補正 + 「時間に見合う成果が出たか」の妥当性振り返り）

15 分で完結するキャップ。起動方式は **default opt-in**（ユーザーが明示的に開く）。自動起動は採用しない（§1.11 「介入しない」原則と整合）。

timer 中は AI が黙る代わりに、棚卸しで AI が能動的に語れる場を確保する非対称構造。

### 1.13 蓄積後インサイト面（Wrapped）

差別化の核（深層）を見せる場。週次 / 四半期 / 年次のナラティブを AI が生成する。

- 「今週最も集中した時間帯」「最もエネルギーを注いだプロジェクト」「先送りされたタスク」
- 「今年最も集中した時間帯」「成長した分野 / 苦手分野の変化」

3 ヶ月後・半年後・1 年後にしか出せない深さで、Motion 等の競合がコピーできない独自軸。

### 1.14 3 つのビュー

**Stack View（今）** — 「今これをやる」を timer と stack top で見せる。timer 動作中は AI が黙る、静かな画面。

**朝の棚卸し画面（毎朝）** — 昨日の振り返り + 今日の意図 + Recovery Score。AI が能動的に語る場。

**Tree View / Wrapped（過去）** — git log tree のメタファーで過去の活動を振り返る + 週次 / 四半期 / 年次の AI ナラティブ。深層の蓄積価値を見せる。


## 2. 情報構造

### 2.1 階層モデル

```
main（自分）
├── project: 転職活動
│   ├── epic: Dirbato選考
│   │   ├── story: 最終面接準備
│   │   │   ├── task: 志望動機パターンA作成 (15min)
│   │   │   ├── task: 企業研究メモ更新 (20min)
│   │   │   └── task: 逆質問3つ整理 (10min)
│   │   └── story: オファー条件交渉
│   └── epic: ULS検討
├── project: SLO推進
└── project: 負荷試験環境構築
```

実運用は project + task の 2 階層を基本とし、epic / story はオプショナルな中間層。

### 2.2 タスク

- **title**（必須、思いついた瞬間に投入）
- **body**（推奨、メモ書き欄。AI が後追い補完の起点に使う）
- **completion_criteria**（Goal / Done / First step、AI 補完、§1.6 / §1.7）
- 見積もり時間: 分単位
- 補正後見積もり時間: 過去実績から自動算出（§1.8）
- 依存イベント: カレンダーイベントへのリンク（任意）
- プロジェクト: 所属プロジェクト
- 分解元タスク: AI 分解で生まれた場合、親タスクへのリンク

### 2.3 タスクのステート

```
idle → active → paused → active → done
```

UI 動詞は **start / stop / complete の 3 つだけ**（[ADR-0058](../adr/0058-timer-three-verbs-and-no-ai-interruption.md)）:

- **start**: idle → active。timer 開始。stack top が current task として bind
- **stop**: active → paused。理由は記録しない（1 タップ主義）。後で朝の棚卸しで振り返り
- **complete**: active → done。実作業時間記録 + stack 2nd が新 top に昇格
- **1-tap interrupt**: active → paused + interrupt event 記録（[ADR-0059](../adr/0059-one-tap-interrupt-recording.md)）
- **calendar auto-stop**: 予定開始時刻に active → paused。event 終了後は自動 active 復帰しない（[ADR-0060](../adr/0060-calendar-event-asymmetric-stop.md)）

各遷移で記録するコンテキスト:
- 遷移時刻
- 直前のイベント（MTG 直後かどうか）
- 時間帯
- 曜日

active 区間の合計 = 実作業時間。この実作業時間と見積もりの乖離が、見積もり補正エンジン（§1.8）の入力データになる（Goodhart リスクへの配慮は §1.8 参照）。

### 2.4 行動ログ

タスクのステートマシンとは別に、以下の行動データを記録する。これが kozutsumi の真のデータ資産。

- スタック内での位置変更（並べ替え）
- スタックからの除外（削除）
- タスクタイトルの書き換え
- 完了条件（Goal / Done / First step）の編集差分
- タスクの統合・再分割
- 分解されたタスクの削除・修正
- 提案されたスタック順序 vs 実際の消化順序
- 割り込みの発生タイミング
- 朝の棚卸しでの事後補正値

### 2.5 イベント

カレンダーから取得する時間固定の予定。

- 開始時刻・終了時刻
- タイトル
- 会議 URL（Meet / Zoom、自動判定）
- 添付資料の有無
- 関連プロジェクト（任意）
- 説明文（Markdown）


## 3. 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| フロントエンド | Next.js | Web アプリとしてスマホ・デスクトップ両対応 |
| 認証 | Google OAuth | カレンダー API 連携の OAuth トークン共用 |
| カレンダー連携 | Google Calendar API v3 | OAuth 共用で企業 Workspace も取得可能 |
| AI サジェスト | Gemini 2.5 Flash | 無料枠で十分（10RPM、250 リクエスト/日） |
| バックエンド/DB | Supabase (Postgres + Auth) | 個人ツールとして Auth + DB で十分 |
