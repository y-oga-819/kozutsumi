# kozutsumi Vision

## このプロダクトは何か

kozutsumiは、使い続けるほど自分に最適化される個人特化AI秘書システムである。最初のアプリケーションはタスク管理。

タスク管理ツールとしての即時的な価値（表層）を提供しつつ、日常的な利用の中で蓄積される行動データからユーザー固有の思考パターン・判断傾向・回避パターン・エネルギー変動を学習し、「自分専用の秘書」を育てる（深層）。

ユーザーは「AIを育てている」という自覚を持つ必要はない。普通に便利だから使っていたら、いつの間にか提案の精度が上がっていて、「なんか最近いい感じに回ってるな」と感じる。これがkozutsumiの目指す体験。

まず自分自身が日常的に使う道具として作る。それが本当に便利であれば、プロダクト化を検討する。


## 誰のためのものか

自分自身。複数のプロジェクトを並行して進めるソフトウェアエンジニアで、以下の特徴を持つ。

- 1日に4〜6件のMTGやイベントが散在する
- 転職活動、技術基盤改善、チーム運営など性質の異なるプロジェクトが並走する
- タスクの粒度がバラバラ（5分のSlack返信から数日の設計作業まで）
- AI（Claude Codeなど）を活用した並列作業を日常的に行う
- 判断をAIに委任したいが、まだAIを信じきれていない段階

加えて、以下の行動特性を踏まえて道具側を設計する（[#234](https://github.com/y-oga-819/kozutsumi/issues/234) の調査シンセシスを参照）:

- **過集中で時間を忘れる** — はまったタスクには長時間没入できるが、終わりが見えない
- **興味の有無で進捗が極端に変動** — 興味の薄いタスクには手が動かない
- **ゴールが見えないタスクで詰まる** — 「何を仕上げれば done か」が曖昧だと freeze する
- **強制的な時間制限で集中が切れる** — Pomodoro 型の強制中断は逆効果


## 何が課題なのか

### 既存のタスク管理ツールはなぜ機能しないのか

既存ツール（Todoist、Linear、Notion、カンバンボード等）は「タスクの一覧管理」には優れている。しかし「今、何をやるべきか」に対する答えを提示してくれない。

リスト型UIの本質的な問題は「全部が見えてしまう」こと。20個のタスクから毎回選ぶ判断コスト自体が生産性を下げている。

### スタック型のタスク管理はなぜ挫折するのか

過去に何度も挫折してきた。原因は3つある。

**原因1: 時間固定のイベントとの衝突。** 14:00に面接があるのに、スタックの上に「Locustシナリオ実装（2時間）」があったら破綻する。

**原因2: 割り込みへの対応コスト。** 急なタスクが入るたびにスタックを再構成する必要がある。

**原因3: 粒度の混在。** 大きなタスクが上にあると着手のハードルが高くなり、先送りされる。

### 根本原因

3つの原因の根底にある構造的問題は、「タスク」と「イベント」と「時間」が未分離のまま1つの仕組みに押し込められていること。

### 今日だけを見ていては足りない

5日後の面接準備を今日から着手しなければ間に合わない。しかし面接対策ばかりでは本業が溢れる。このバランス判断を毎朝手動でやるのは認知負荷が高すぎて続かない。

### 人間の見積もりは外れる前提で道具を設計する

ソフトウェア開発の歴史が何十年もかけて証明してきた事実。自分が「20分で終わる」と思ったドキュメント作成は実際には44分かかる。これは「人間が悪い」のではなく構造的な認知バイアスで、**ユーザーを変えるのではなく道具側で吸収する**（借金玉「自分を変えるな、道具に頼れ」/ KC Davis "morally neutral"）。

苦手なタスクほど見積もりが甘くなる傾向自体は、kozutsumi の **行動データ蓄積** の格好の材料になる。


## 差別化の核（モート）

kozutsumi の本質的な差別化は **2 層構造** にある（[ADR-0057](./adr/0057-redefine-moat-as-goal-driven-ai-decomposition.md)）。

### 表層 — ゴール駆動の AI 分解

でかいタスクをそのまま放置せず、**1 時間単位でゴールを追いかけられる粒度** に AI が分割し続ける。各子タスクには Goal / Done / First step が明文化される（[ADR-0061](./adr/0061-ai-decomposition-one-hour-target-and-done-condition-schema.md)）。

Motion との対比:

- **Motion**: 60 分タスクをそのままスケジュールする
- **kozutsumi**: 60 分に **到達できる単位** に砕いてからスケジュールする

これが「ゴールが見えないタスクで詰まる」想定ユーザー特性への直接処方になる。

### 深層 — 行動データ蓄積による分解精度の個人最適化

タスク管理の日常利用で蓄積される **着手パターン / 中断パターン / 回避パターン / エネルギー変動 / タスク種類ごとの見積もり誤差** が、AI 分解の精度を上げる学習素材になる。Motion 等の競合は属性ベース（優先度・締切・所要時間という静的な属性）でスケジュールするが、kozutsumi は **行動ベース** で分解精度を育てる。

この違いは蓄積データの構造が根本的に異なることに起因する。競合が「機能を真似する」だけでは追いつけない。なぜなら行動データを取得するためのデータ設計思想が違うから。

3 ヶ月後、半年後、1 年後にしか出せないインサイト（週次 / 四半期 / 年次 Wrapped、朝の Recovery Score 等）は、本質的に「データ蓄積後にしか出せない深さ」であり、**機能比較に巻き込まれない独自軸**になる。

### UX 原則 — 「ユーザーが触るのは timer の 3 動詞だけ」

差別化の核は表層・深層だが、それを支える UX 原則として **start / stop / complete の 3 動詞**（[ADR-0058](./adr/0058-timer-three-verbs-and-no-ai-interruption.md)）を死守する。

- 過集中を強制中断しない（[ADR-0063](./adr/0063-no-intervention-on-hyperfocus.md)）
- タスク作成は title 必須のみ（[ADR-0064](./adr/0064-task-creation-title-only-with-ai-template-fill.md)）
- AI 介入は timer 文脈外（タスク詳細画面 / 朝の棚卸し）でのみ発動
- 突発割り込みは 1-tap で記録（[ADR-0059](./adr/0059-one-tap-interrupt-recording.md)）
- Calendar event は非対称 stop（[ADR-0060](./adr/0060-calendar-event-asymmetric-stop.md)）

これは想定ユーザーの行動特性（過集中 / 強制時間制限で集中が切れる）と矛盾しない設計を成立させる前提。

**この前提は全ての設計判断の基盤になる。** 機能レベルで競合と比較してはいけない。常に「行動データを取れるか」「蓄積が意味を持つか」「ユーザーを変えずに道具で吸収するか」を基準に判断する。
