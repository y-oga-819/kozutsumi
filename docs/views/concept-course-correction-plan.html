kozutsumi コンセプト軌道修正計画

想定ユーザー (= 自分) の行動特性を踏まえた 4 領域の調査 (#234) を受けて、現 vision の中核機構 3 つのうち 1 つを再設計し、独自軸 (AI 分解) と新たな儀式 (朝の棚卸し) を中心に据え直す方向の計画。

TL;DR

差別化の核を 「行動ベース」 から 「ゴール駆動の AI 分解 (= 1 時間粒度の到達点に砕く)」 に書き換える。「行動ベース」は AI 分解の精度を上げる学習素材として深層に残す。

UX は 3 動詞 (start / stop / complete) + 割り込み 1-tap + calendar 非対称 stop に統一。タイマー駆動の active / paused state machine は維持するが Autonomy 強化方向にアレンジ。

朝 15 分の棚卸し を新たな儀式として組み込み、行動データ蓄積後のインサイト提示 (Whoop 型) で「秘書らしさ」を見せる場にする。過集中は 介入しない (パフォーマンスが出ているなら続行)。

既存 ADR はほぼ supersede 不要。新規 ADR 5〜7 本で確定する。Vision の書き換えは ADR 確定後。

01

Why now

想定ユーザー (= 自分) の行動特性が、現 vision の中核機構と矛盾する可能性が浮上したため。

想定ユーザーには次の 4 特性がある (#234 本文より):

  1. 過集中で時間を忘れる
  2. 興味の有無で進捗が極端に変動
  3. ゴールが見えないタスクで詰まる
  4. 強制的な時間制限で集中が切れる

一方、現 vision / architecture の中核機構は次の 3 つ:

  • per-task 予実計測 (見積もり補正エンジン)
  • AI 自動分解 (分解挿入による着手ハードル低減)
  • タイマー駆動 active-paused state machine

特性 (1)(4) と「タイマー駆動」、特性 (2) と「per-task 予実計測の Goodhart リスク」、特性 (3) と「分解の意味付け」がそれぞれ衝突しうる。これを検証するため #234 で 4 領域 (神経多様性向けタスク管理 / 行動経済学 / ゲーミフィケーション / 異分野報酬設計) の調査を実施し、原文を comment 1〜4 に保管した。本文書はその統合シンセシスを踏まえた軌道修正計画。

02

Context

関連する現状と、決定の前提として参照する既存 ADR / docs。

現 vision の差別化軸

現 vision.md L56-66:「Motion 等の競合は 属性ベース (優先度・締切・所要時間) でスケジュール、kozutsumi は 行動ベース (着手 / 中断 / 回避 / エネルギー変動) でスケジュール」と書いている。

調査 4 本横断のシンセシス (要点)

  • AI 分解 / ドリルダウン型ゴール体験: 4/4 本が強く支持 (Solanto / Mahan / 小鳥遊 / Davis / Khan Academy)
  • per-task 予実計測: タスクシュートと理念上同質で支持。ただし能動入力よりサンプリング / 受動センシング寄りが ADHD 文脈では推奨 (ESM, on-device ML)
  • タイマー駆動: 3/4 本で「逆効果」警告。Wall of Awful (Mahan) 状態に強制開始は freeze→fight/flight、Duolingo Heart 型は「最も非倫理的」
  • 蓄積後インサイト提示 (Whoop / Spotify Wrapped 型) は「ユーザー定着の単一最強機能」と評価される独自軸
  • 罰しない / 失わせない (Finch / Apple Rings) が ADHD 層に支持される。Habitica / Duolingo は反面教師
  • co-regulation (body doubling) は AI に体がないため完全代替不可。kozutsumi は「行動パターン理解 + 外部記憶補完」で勝負する方が筋が良い

03

Options explored

シンセシスを受けて検討した 4 つの方向性。

Option a — 現状維持 + ドッグフードで判断

3 中核機構をすべて維持。実利用で逆効果か検証。
長所: 投資保全、判断を遅らせられる。
短所: 検証期間中の体験が悪化する可能性。学術的に既知のパターン (Wall of Awful, Heart 型強制中断の害) を再発見するだけになる懸念。

Option b — 最小侵襲な軌道修正 ★

タイマー駆動を「3 動詞 + Autonomy 強化」にアレンジ、AI 分解を差別化軸の中心に据え、朝の棚卸しを儀式として追加。per-task 予実計測は維持 (能動入力で取れるシグナルは残す)。
長所: 既存 ADR をほぼ supersede せずに済む。差別化軸が「Motion: 60min そのままスケジュール / kozutsumi: 60min に到達できる単位に砕いてからスケジュール」と明快化する。
短所: 「3 動詞」UI が地味で蓄積価値が見えにくい → 朝の棚卸し / Wrapped で補う。

Option c — 観測専用に倒す

タイマー / per-task を観測専用にし、UX の中心を「ドリルダウン型ゴール体験 + Wrapped 型インサイト」に置き換え。
長所: Autonomy 最大化、Goodhart リスクほぼゼロ。
短所: 既存 ADR-0003 / 0004 / 0024-0026 を広範に supersede。リプレース工数大。

Option d — Quest Log / Skill Tree 寄りに転換

event-driven stack を捨て、Quest Log (WoW) / Skill Tree (Khan Academy) 寄りの構造へ。
長所: ゲーミフィケーションを vision レベルで取り込める。
短所: 実質「別アプリ」。投資保全がほぼゼロ。

05

Decision

確定した 6 つの判断。各々を ADR に distill する (§09 Open Questions の最後に ADR 候補リスト)。

差別化軸の再定義

Motion は 60 分タスクをそのままスケジュールする。kozutsumi は 60 分に 到達できる単位 に砕いてからスケジュールする。

D1 · 差別化の核を「ゴール駆動の AI 分解」に書き換える

表層 (体験): でかいタスクを 1 時間単位でゴールを追いかけられる粒度 に分割し続ける。
深層 (基盤): 行動データ蓄積による分解精度の個人最適化 (現「行動ベース」はここに残す)。

D2 · UI 動詞は 3 つだけ — start / stop / complete

ユーザーが触る動詞はこの 3 つに統一。stack top が timer の current task として自動 bind。「次に何をやるか」を選ぶ操作はゼロ。

Note. Things 3 / Bear の Quiet Productivity と同方向。Streak / Heart 系の罰デザインを構造的に避けられる。

D3 · 割り込みは 1-tap で記録 + timer 停止

Slack / Notion / huddle 等の突発割り込みは、専用ボタン 1 タップで timer 停止 + 割り込みイベント記録。事前分類はしない (摩擦最小化、後でまとめて振り返る)。

D4 · Calendar event は 非対称 stop

予定の開始時刻になったら timer 自動停止。予定が終わっても 自動再開しない — ユーザーが明示的に start を押すまで何も起きない。Externalize time (Barkley) と Autonomy 保持 (SDT) の両立。

D5 · AI 分解の発火条件 = 「ゴールが追いかけられる粒度」推奨制

論点 A=a2 / B=b3 で確定:

  • 1 時間 = 推奨目安 (強制ではない)。AI が 30-90 分レンジを目指すが、5 分や 3 時間も許容。ユーザー調整 OK
  • 各子タスクには 完了条件を AI が言語化 (Solanto "first step is too big" / Davis "First steps")
  • 並行して、親タスクの達成度を子の積み上げで 可視化 (ParallelogramProgress 系の延長)
  • 発火: stack top に start を押せない状態 (= dwell time / 着手不能シグナル) を検出した時の 提案。強制しない

D6 · 朝 15 分の棚卸し儀式

論点 C=c2 で確定。中量版:

  • 昨日の summary (完了 / 中断 / 割り込み件数 + 行動データの示唆) — Whoop Recovery Score 風
  • 今日の stack 確認 + 重い順 / 軽い順並び替え
  • 「今日の意図」を 1 つだけ選ぶ

15 分で完結するキャップ。c3 (週次振り返り / 先送り判断) は別セッションに分離する想定。

D7 · 過集中は介入しない

特性 (1) 過集中で時間を忘れる、への対処は しない。パフォーマンスが出ている時間を強制中断する害 (Comment 1: Wall of Awful / Comment 3: Heart 系) のほうが大きい。Hallowell VAST「right kind of difficult」と整合。

Caution. 半熟ポモドーロ (Comment 2 DASHI) の「乗っていても止める」原則は採用しない。過集中の燃費悪化より、強制中断の Autonomy 剥奪のほうがコストが高いと判断。
06

Mechanism

「3 動詞 + 1-tap 割り込み + calendar 非対称 stop + AI 分解」のデータ / 制御フロー。

solid = sync action, dashed clay = system-triggered stop, dashed olive = AI proposal

状態遷移の補足

  • start: idle → active。timer 開始。stack top が current task として bind
  • stop: active → paused。理由は記録しない (1 タップ主義)。後で振り返り
  • complete: active → done。実作業時間記録 + stack 2nd が新 top に昇格
  • 1-tap interrupt: active → paused + interrupt event 記録。stack 構造は変えない (= 割り込みタスクは stack に push しない)
  • calendar auto-stop: 予定開始時刻に active → paused。event 終了後は自動 active 復帰しない (= ユーザー start 必要)
  • AI decompose proposal: stack top が dwell ≧ N 分かつ start 押されない時にトリガー (シグナル設計は ADR で詰める)
07

Milestones

確定方針を体験単位で 4 つの milestone 候補に分解。順序は持たない (kozutsumi-flow §1)。既存 M9 / M10 との整理も含む。

M-α

3 動詞 UX への統一

start / stop / complete + 1-tap interrupt + calendar 非対称 stop。既存 ADR-0003 / 0004 を拡張する形で実装。新規 ADR: D2 / D3 / D4 を distill。

M-β

ゴール駆動の AI 分解 (1h 推奨制 + 完了条件 + 親進捗)

D5 を実装。既存 M9「大量に分解されたタスクの進捗体験」を scope 書き換え or close + 新 milestone 起票 で吸収。M10「秘書から相談があります」は AI 分解 proposal の延長として M-β の中に取り込めるか要検討。

M-γ

朝 15 分の棚卸し儀式

D6 を実装。Whoop 風 Recovery Score の表示、stack 並び替え、今日の意図選択。新規ビューが要る (現 stack view とは別画面)。

M-δ

蓄積後インサイト面 (週次 / 四半期 / 年次 Wrapped)

差別化軸 (深層) を見せる場。Spotify Wrapped 型ナラティブ生成。M-γ で揃ったデータが土台になるので順序的には後ろ。

既存 milestone との関係

  • M8 Googleカレンダー連携強化 — D4 (calendar 非対称 stop) と関連するが直接の影響なし。as-is で並走可
  • M9 大量に分解されたタスクの進捗体験 — M-β に吸収。description を書き換え or close + 新 milestone 起票
  • M10 AI秘書との対話によるより精度の高いタスク分解 — M-β の AI 分解 proposal で多くを吸収。「対話」部分が独立価値あるなら別 milestone のまま、なければ統合

Risks & Mitigations

軌道修正に伴い壊れうる箇所と対策。

stack top に行きたくないタスクが居座って start を押せない (Wall of Awful)
high
D5 の AI 分解 proposal がここを拾う設計。dwell time / 着手不能シグナル検出時に「分解する？ 入れ替える？」を提案。"permission to start" 文言 ("5 分だけ") も用意。
過集中 (D7 で介入しない判断) で stop し忘れて翌朝まで timer が回り続ける
med
非アクティブ検知 (キーボード / マウス無入力 N 分) で auto-pause → 戻ってきた時に「これいつまで？」を 1 タップで補正できる UI。介入ではなく事後補正。
per-task 予実計測 KPI (見積もり精度 ±20%) が Goodhart's Law で「測定が目標化」
med
feature-spec.md の KPI 節を「精度」中心から「定着 + インサイト精度」中心に組み替える。LeetCode 型 (S/M/L 別カウント) を参考に。Vision 書き換えと同時。
3 動詞 UI が地味すぎて「秘書らしさ」が消える
med
M-γ (朝の棚卸し) と M-δ (Wrapped) で「秘書らしさ」を見せる場を担保。stack 画面は静か、棚卸し画面で語る、の二層構造。
ADR の supersede 判断が後手に回り、コードと ADR が乖離する
low
本文書の Decision を ADR に distill する作業を M-α 着手前に完了させる (kozutsumi-adr skill 経由)。本 HTML は ADR 化したら削除。
co-regulation を AI で代替できない問題が UX 体感に出る
low
そもそも代替を狙わない。「行動パターン理解 + 外部記憶補完」で勝負する方針を vision に明記する (vision 書き換え時)。
09

Open Questions

まだ決着していない論点。決着したら ADR に distill して本文書から削除。

Q1. AI 分解 proposal の発火シグナル設計
「dwell time N 分 + start 未押下」が第一候補だが、N の値、他のシグナル (タブ切替頻度 / scroll 量 / 過去類似タスクの先送り回数) を組み合わせるか。decide-with: M-β 着手前の ADR で確定
Q2. M10「秘書から相談があります」を独立 milestone として残すか
D5 の AI 分解 proposal で「対話」の入口は作れる。さらに「複数往復のヒアリング」が独立価値を持つかは要検討。decide-with: M-β 設計時
Q3. KPI の組み替え方針 (feature-spec.md L50-69)
「見積もり精度 ±20% / トップ開始率 80% / スタック消化率 70%」を「定着 + インサイト精度」中心に組み替える。LeetCode 型 (S/M/L 別カウント) を参考に。decide-with: vision 書き換えと同時
Q4. vision.md の差別化軸書き換えタイミング
D1〜D7 を ADR に distill 完了してから vision.md を書き換える順序。HTML 上で先行する案を提示するのは可。decide-with: ADR 5〜7 本確定後
Q5. 過集中の事後補正 UX (Risk: stop し忘れ)
非アクティブ検知 N 分の閾値、補正 UI の形 (modal / inline)、補正された記録は実績データとしてどう扱うか (見積もり補正エンジン入力に含めるか除外か)。decide-with: M-α の β 期間に dogfooding で観察
Q6. 朝の棚卸しを「儀式」として強制するか opt-in か
Comment 4「朝の Task Recovery Score」は提示型 (説教調を避ける)。kozutsumi では「起動時に自動表示」or「ユーザーが開く」のどちらか。前者は習慣化に有利、後者は Autonomy 寄り。decide-with: M-γ 設計時
Q7. 既存 stack view と棚卸し画面の関係
stack view (静か / current task に集中) と棚卸し画面 (語る / インサイト + 並び替え) の二画面構成。導線、デフォルト起動画面、相互の遷移パターン。decide-with: M-γ 設計時

distill 候補 ADR (M-α 着手前)

  1. 差別化の核を「ゴール駆動の AI 分解」に再定義 (D1) — vision-level、現「行動ベース」の位置付け再整理
  2. UI 動詞を 3 つに絞る + stack top auto-bind (D2) — ADR-0003 / 0004 の延長
  3. 1-tap 割り込み記録 (D3) — 既存の割り込み push/pop モデル (architecture 1.4) の supersede or 補強
  4. Calendar event は非対称 stop (auto-stop / manual resume) (D4)
  5. AI 分解は 1 時間推奨制 + 完了条件付き + 着手不能シグナル発火 (D5) — ADR-0017 / 0049 / 0053 の延長
  6. 朝 15 分の棚卸し儀式 (D6) — 新画面、新 entity (daily-review)
