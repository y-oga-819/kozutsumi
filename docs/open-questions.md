## Stack View カードの情報設計

ADR 0016 (Stack View は子のみフラット表示) を Accepted で確定させたが、Vercel preview で A〜D を実際に触った結果、A の「親子関係 / 完了境界が見えない」弱点が想定より重いと判明し、ADR を Proposed に戻して再議論中。

C (折りたたみ) と D (breadcrumb) の長所を取り込んだハイブリッド案 (Variant E) を作って再検討する。直接案を作る前に、**Stack カードに何を表示するか / それは何のために必要か** を以下に整理する。E はこの整理を満たすレイアウトとして設計する。

**確定している前提:**

- Stack 行 = 子のまま (ADR 0016 で確定済みの方向性。C 案の linearity 崩壊は採らない)
- ADR 0017 (非同期分解) / ADR 0018 (`parent_task_id` 保持) は変えない

**情報設計の論点 (E 案で解決すべき要件):**

| 情報 | なぜ必要か | 触ってみて見えた根拠 |
|---|---|---|
| 親タスク名 (= ゴール) | 子だけ見て「何のため?」が即答できる | A: 「志望動機パターンA作成」だけだと文脈不明 |
| 親の完了境界 (子何個でゴール到達 / 残り何個か) | 「当初やりたかったこと」が終わる感覚を持てる | A: 何を消化すれば親が完了か見えない |
| 親由来のデッドライン (`dep event`) を子から見せる | 「明日 14:00 までに親を完了させるには、どの子を全部やる?」が見える | D: 親が分解されるとデッドラインが消える |
| 親グループの合計時間 | 全体の残り工数感がつかめる | C: 唯一これが見えていた |
| Top カードの視覚的優位性 (情報量 + 目立ち方) | 「次の 1 つ」が一目で分かる | D: top と他の行が均質で目立たない |
| Top 以下のカードの情報絞り込み | 高密度すぎると疲れ、Top との対比も崩れる | D: 全カードに breadcrumb が乗って情報過多 |
| 同親グループの連続性は必須でない | DnD 並び替えで分断されて当然、まとまりを強調しすぎない | B: 縦線が分断されると意味が薄れる |

**進め方:**

`docs/design/` に長い設計書は作らず、上の表だけを論点として `src/features/stack-view/__experiments__/VariantE.tsx` に直接組み込む。プロトタイプを Vercel preview で触り、固まったら ADR 0016 を E 案で書き直して Accepted にする。

決着したら本セクションを削除する。

## 行動データのプライバシー

行動ログには個人の思考パターンや弱点が含まれる。プロダクト化時にはデータの所有権・削除権・暗号化が重要な設計要件になる。個人ツールの段階ではSupabaseのRow Level Securityで対応。
