## AI 分解後のタスク登録 / 表示方式

Phase 3 の最大の設計論点。AI がタスクを子タスクに分解した結果を、kozutsumi の Stack View にどう積むかという体験設計。

**確定している方針:**

- **同期 vs 非同期 → 非同期**。タスク追加と AI 分解は一気通貫にしない。元タスクが即座にスタックに乗り、AI 分解結果は後追いで反映される。「気づいたら細かくなってる」体験 (vision「育てている自覚を持たせない」) に寄せる。
- **親タスクの扱い → 残す**。`tasks.parent_task_id` は schema に既にある。データ構造としては parent-child を保持し、Tree View には残る。

**未決着 (Phase 3 着手前にプロトタイプで体験比較したい):**

親を「データ上は残す」前提で、Stack View にどう表現するかが核心。Stack の linearity (「次の 1 つ」が明確であること) と親子の構造表現が衝突する。候補:

- **A. 子のみスタック / 親は Tree View だけ**
  - スタックは分解済みの子だけがフラットに並ぶ。親はデータ上残るが Stack View には出ない
  - 体験は「タスクが勝手に細かくなる」になる。スタック linearity が一番素直
  - 親のコンテキスト（何のための一連か）が Stack 上で見えなくなる
- **B. 子フラット + 親バッジ / グループ化**
  - 子がフラットに並びつつ、各子カードに親名を小さく表示。連続する同親の子は色 / 縦線でグルーピング
  - コンテキスト維持と linearity の両立を狙う。賑やかになりやすい
- **C. 親をスタックに残し、展開で子を表示 (折りたたみ式)**
  - スタックの 1 行 = 親、展開で進捗 / 子の状態が見える
  - 親子関係が UI で明示される。「次にやるのは親 or 子」の意味が曖昧化するリスク
- **D. breadcrumb 表示**
  - スタックトップに「dirbato 面接準備 > 志望動機」のようにパス表示
  - linearity + コンテキスト両立。多階層 (project → epic → story → task) で長くなる

**比較軸:**

- vision との整合（「気づいたら細かくなってる」「育てている自覚を持たせない」）
- Stack View の linearity（次の 1 つが明確か）
- 親の存在感 / コンテキスト保持
- AI 分解品質が低い時のフォールバック容易性（人手で再統合 / 削除しやすいか）
- 暗黙的フィードバック収集のしやすさ（削除 / 統合 / 再分割の操作が action_log に取れるか）
- Tree View（過去ログ）との整合

**進め方:**

実装に走る前に複数案をプロトタイプ / モックで作って体験比較する。コアの体験なのでここを急がない。

決着したら ADR 0016 として起票し本セクションを削除する。1 / 2 (非同期 / 親残す) も併せて ADR 化する想定。

派生する判断（UI が決まれば自動で決まるもの）:

- 子タスクが親の `project_id` / `depends_on_event_id` / 見積もり継承をどうするか
- AI が「分解不要」と判断した場合の挙動
- 分解結果到着前に親が active 化された時の race condition
- `task_decomposed` / `decomposition_modified` 等の新 ACTION_TYPE 追加判断

## 行動データのプライバシー

行動ログには個人の思考パターンや弱点が含まれる。プロダクト化時にはデータの所有権・削除権・暗号化が重要な設計要件になる。個人ツールの段階ではSupabaseのRow Level Securityで対応。
