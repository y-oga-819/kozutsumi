### その他構造的気付き

- **「現状継続」型 ADR**: 0028 のように「既存パターンを続ける」判断を ADR 化するのは、他選択肢の評価記録として価値がある一方、内容が薄くなる傾向。本リポジトリでは数が少なく問題化していない。
- **方針宣言型 ADR**: 0013 / 0036 のような「不変条件 / 上位方針」型 ADR は supersede が事実上難しく、判断 ADR と性格が違う。0013 は vision 直結で価値が高いが 0036 はやや弱い。
- **post-Accept 改訂の蓄積**: 0023 を典型に、0011 / 0035 でも本文に「Issue #N で確定」「migration 指示」など Accepted 後の編集と読める節がある。skill §5 を厳格に運用する場合は supersede 単位を細かく刻む必要がある。

## 全体所感

44 本中 30 本が健全、14 本が borderline、弱い ADR は 0 本という構成は、運用ルール (skill `kozutsumi-adr`) が機能していることを示している。borderline の多くは「skill §5 (Accepted 本文不変)」と「skill §1 (1 判断 = 1 ADR)」の境界事例で、ADR 化したい設計判断が深く構造化されている (0016 の Stack View / 0023 の migration diff CI / 0034〜0035 の calendar lifecycle) ほど 1 ADR に sub-decision が膨らむ傾向が見える。番号衝突は merge 取り込み (commit ce87a53) で、ADR 0031 の broken references は本 audit commit で fix-references 済。残る取り組みは 0023 の改訂蓄積を split + supersede で整理し、0016 / 0035 の split を milestone 計画タイミングで検討する程度。Phase 語彙は廃止されたが、既存 ADR 本文への遡及修正は skill §5 と衝突するため grandfather 扱いが妥当で、新規 ADR から milestone 語彙に揃えていく運用で十分。
