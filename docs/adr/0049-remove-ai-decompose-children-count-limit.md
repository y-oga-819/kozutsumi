# ADR 0049: AI 分解の件数上限を撤廃する

- **Status**: Accepted
- **Date**: 2026-05-06
- **Related**: `docs/design/vision.md` / Issue #215 / Issue #166 / Issue #208 / [ADR-0017](./0017-ai-task-decomposition-async.md)

## Context

現状の AI 分解は `src/shared/ai/prompts/decompose.ts:40-41` で `MIN_CHILDREN=2 / MAX_CHILDREN=7` という保守的な閾値を持ち、prompt 上でも「件数は X〜Y 件」と AI に指示している。これは Phase 1 / 2 で「異常な大量出力を避ける」ための初期ガードだったが、実運用が進むにつれて以下のミスマッチが顕在化した。

- 章立てのある本（「○○の本を読む」を 8 章単位で割りたい）/ 複数手順タスク / 旅程組み立てなどで、上限 7 が支配的に効き、AI が「自然な分け方」を出せない
- 結果として user が手動で resplit して件数を増やすことになり、AI 分解の価値が下がる

milestone 9「大量に分解されたタスクの進捗体験」では「数十件規模の分解を許容する」という前提に立ったため、件数上限の撤廃が必要になった。

なお、上限を「タスク種別ごとに持つ」という案もあったが、開発者が `読書 / 学習 / 執筆 / 開発` のような静的カテゴリを定義する形は vision.md の差別化軸「行動ベース（属性ベースではない）」と矛盾するため採らない（roadmap 親 #208 の議論）。

## Decision

AI 分解の件数上限 (`MIN_CHILDREN` / `MAX_CHILDREN`) を撤廃し、件数の自然さは AI の判断に委ねる。prompt の件数指示は「自然な単位で分解する」に置き換え、アプリ側で件数を理由に reject しない。

## Consequences

### 肯定的影響

- 章立て・手順タスクなどで AI が自然な粒度（数十件規模）で分解できる
- 「件数は X〜Y 件」という静的な属性指示が消え、vision「行動ベース」に近づく
- user が手動で resplit する負担が減り、AI 分解の体感価値が上がる

### 否定的影響・トレードオフ

- 異常な大量出力（誤って数百件など）のガードが消える。発生頻度は低い前提で受け入れる
- 大量分解時の UI 破綻（`ParallelogramProgress` が 10 件超で破綻）が顕在化する。これは別 ADR / 別 issue (#166) で扱う
- AI の出力件数が安定しないことを前提にした他箇所のロジック（テスト、UI レイアウト想定）を見直す必要がある

## Alternatives considered

- **案 A: 上限を 20〜30 件に緩和**: 結局静的閾値で、章立て次第ではまだ足りない。撤廃と緩和の中間に意味のある根拠がない → 不採用
- **案 B: タスク種別ごとの上限**: 「読書 / 学習 / 執筆 / 開発」のような種別を持ち上限を変える。開発者がカテゴリを管理する形になり vision「行動ベース」と矛盾する → 不採用（議論の保管庫として #211 に残す）
- **案 C: AI が大量出力したらアプリ側で trim**: AI の判断を尊重せず、件数で切り捨てると意味的に分解として壊れる → 不採用
- **案 D: アプリ側自己ループで再分解**: 大粒度子タスクに対して自動 re-decompose を回す。本 ADR の対象ではなく独立した手段。roadmap 子 #212 で別途検討 → 本 ADR スコープ外

## Notes

- 異常出力（数百件超など）の検知が将来必要になったら、その時点で別 ADR を起票する（事前最適化しない）
- 大量分解時の UI 表示戦略は別 ADR で扱う（本 ADR は「件数上限を外す」判断のみ）
