# ADR 0003: イベント駆動スタックをコア体験の中心に据える

- **Status**: Accepted
- **Date**: 2026-04-15
- **Related**: `docs/design/vision.md` / `docs/design/architecture.md` §1.2-1.4 / `docs/design/competitive-analysis.md`

## Context

タスク管理ツールには大きく3系統のメタファーが存在する。

1. **リスト型**（Todoist / Linear / Notion / カンバン）: 「全部が見える」。
   判断コストが高い。「今何をやるか」に答えない。
2. **スタック型**（伝統的 GTD ツール）: 上から1つずつ消化。判断コストは低い。
   ただし過去に何度も挫折: (a) 時間固定のイベントとの衝突、(b) 割り込み対応コスト、
   (c) 粒度の混在による先送り（vision.md）。
3. **属性ベース自動スケジューリング**（Motion / Reclaim.ai）: 優先度・締切・所要時間
   という静的属性からカレンダーに配置。

kozutsumi は「個人特化 AI 秘書」として行動パターン分析の深さで差別化する（vision.md）が、
その前提として**日常的に使い続けられる UI のコア**が必要。どのメタファーを基盤に置くか。

## Decision

**イベント駆動スタック** をコア体験の中心に据える。
具体的には以下の構造を Phase 1 から実装する（architecture.md §1.2-1.4）:

- カレンダーイベントをタイムラインに固定表示
- イベント間の空き時間を視覚化
- タスクは空き時間の中でスタックから1つずつ消化
- 優先度順にスタックを積む（空き時間へのフィッティングは副次）
- 割り込みは push/pop で構造的に表現

これにより、スタック型が挫折する3原因を構造的に解消する。

## Consequences

### 肯定的影響

- **判断コストの低さを維持**（スタックのトップだけ見る）しつつ、時間固定イベントとの
  衝突・割り込み・粒度問題を構造的に解消。
- **Phase 1 の UI 検証が通れば即座に日常利用**に移行できる。著者が使い続けることで
  行動ログが蓄積し、Phase 3-4 の学習基盤になる。
- Motion の属性ベース自動スケジューリングとは差別化された体験。kozutsumi は
  「自動配置」ではなく「ユーザーの優先度判断 + AI が次のタスクを提案」という構図。

### 否定的影響・トレードオフ

- **カレンダー連携への依存が構造的**。Phase 2 で Google Calendar 連携を入れるまで、
  イベントは手動入力になる（Phase 1 の体験検証はこの状態で行う）。
- **「ユーザーの優先度判断を尊重する」ことで、ユーザー自身が並べ替えに迷う場面がある**。
  これは Phase 4 の二段階提案で AI が補助することで解消する設計。

### 将来的な制約

- 本決定は kozutsumi の**体験の幹**。アプリの別ドメイン展開（ドキュメント生成支援等の
  横展開構想）の際にも、この「イベント駆動 + スタック」のメンタルモデルを基礎に置く
  可能性が高い。

## Alternatives considered

- **リスト型を採用**: 判断コスト問題を解決できない。vision.md の問題認識と矛盾。不採用。
- **属性ベース自動スケジューリング（Motion 類似）**: 行動パターン分析という
  kozutsumi の差別化と接続が薄い。属性ベースは「何をいつやるか」を決めるが、
  「なぜそのタスクが回避されているか」の分析には向かない。不採用。
- **ハイブリッド（日常はリスト、重要日のみスタック）**: モード切替の認知負荷が高い。
  コア体験が二重化するとパーソナライズの学習データも分散する。不採用。
