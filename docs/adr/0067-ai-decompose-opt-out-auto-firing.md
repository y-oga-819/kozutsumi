# ADR 0067: AI 分解の自動発火は opt-out 可能にし、後追い補完は auto 固定とする

- **Status**: Accepted
- **Date**: 2026-05-16
- **Related**: `docs/design/vision.md` / [Issue #243](https://github.com/y-oga-819/kozutsumi/issues/243) / [Issue #244](https://github.com/y-oga-819/kozutsumi/issues/244) / [Issue #245](https://github.com/y-oga-819/kozutsumi/issues/245) / [ADR-0017](./0017-ai-task-decomposition-async.md) / [ADR-0058](./0058-timer-three-verbs-and-no-ai-interruption.md) / [ADR-0061](./0061-ai-decomposition-one-hour-target-and-done-condition-schema.md) / [ADR-0064](./0064-task-creation-title-only-with-ai-template-fill.md) / [ADR-0066](./0066-decompose-completion-criteria-deliverable-done-first-step.md)

## Context

ADR-0061 と ADR-0064 は Notes で「補完タイミング / ユーザー手動入力との競合解決」を M-β 設計時に確定すると委譲した。本 ADR は Issue #244 として、AI 分解 / 補完の発火モデルと競合解決を確定する (schema の項目は ADR-0066 で確定済み)。

現状: ADR-0017 で AI 分解はタスク作成時に自動発火する。ADR-0064 で AI 後追い補完も auto (詳細画面閲覧時 / 朝の棚卸し) と決まっている。両者とも vision「気づいたら細かくなってる」の受動的体験に沿う。

論点は「この auto をユーザーが止められる必要があるか」。検討の結果、**分解と後追い補完で誤りの影響度が非対称**だと整理した:

- **分解が誤ると**: 不要な子タスクが stack に増え、ユーザーがそれを削除する手間が発生する。削除すべきタスクが増えることは実務フロー (今日のスタックの見通し) に直接影響する。特に AI 分解の性能が未成熟な時期 (dogfooding 初期) は誤分解が多く、コストが無視できない。
- **後追い補完が誤ると**: 完了条件 (deliverable / done / first_step) の文言が多少ずれるだけで、ユーザーは読み飛ばせばよい。stack の構造は変わらず、実務への影響は軽微。

この非対称性から、分解にだけ「auto を止める退路」を用意し、後追い補完は auto 固定とする。

## Decision

1. **分解の自動発火 (ADR-0017) は維持し、デフォルト ON とする。** vision の受動的体験 (「気づいたら細かくなってる」) をデフォルトで保つ。
2. **分解の自動発火に opt-out 制御を足す。** タスク作成フォームに per-task トグルを置き、ユーザー設定でグローバルなデフォルト (ON / OFF) を切り替えられる。分解性能が未成熟な時期や、自動分解を望まない局面で auto を切れるようにする。
3. **手動分解ボタンを置く。** auto を切った (トグル OFF / 設定 OFF) タスクや再分解したいタスクのために、タスクカードから明示的に分解を起動できるボタンを設ける。
4. **後追い補完 (#245) は ADR-0064 のまま auto 固定とする。** 誤っても読み飛ばせて実務コストが低いため、opt-out は設けない。timing は ADR-0064 のとおり (詳細画面閲覧時 / 朝の棚卸し)。
5. **競合解決** (ADR-0064 が委譲した論点): AI 補完はタスクの **未補完フィールドのみ** 書く (ユーザーが手動入力した値は上書きしない)。加えて AI 実行中 (`decompose_status='decomposing'`) は当該タスクの完了条件編集を UI でロックする。

## Consequences

### 肯定的影響

- 分解性能が未成熟な時期でも、auto を切れば誤分解の削除コストを避けられる。dogfooding を回しやすい。
- デフォルト ON なので vision「気づいたら細かくなってる」の受動的体験は保たれる。ADR-0017 / ADR-0064 の前提を覆さず、opt-out の退路を足すだけで済む。
- 誤りの影響度の非対称 (分解は実務コスト大 / 補完は軽微) に制御コストを合わせている。補完にまで opt-out を設けないので、設定 UI と onboarding が単純なまま。
- 競合解決を「未補完のみ書く + 実行中ロック」で閉じられる。ユーザーが書いた値が AI に消える事故が構造的に起きない。

### 否定的影響・トレードオフ

- 分解の発火に per-task トグル + グローバル設定という制御面が増える。作成フォームと設定画面に UI を足す必要がある。
- グローバル設定を OFF にしたまま忘れると、大タスクが分解されないまま放置される。手動分解ボタンで救えるが、押し忘れる経路は残る。
- タスクカードに手動分解ボタンが増える (ADR-0065 の割り込みボタンと同居)。アイコン / 配置の整理が実装で必要になる。

## Alternatives considered

- **分解も補完も全面手動 (デフォルト OFF)**: 一度は採用しかけたが、デフォルト OFF は vision の受動的体験を壊し、ADR-0064 が不採用とした「default オフ」を覆すことになる。誤分解の削除コストは「分解性能が未成熟な開発期の問題」であって、opt-out が存在すれば足りる (デフォルトまで OFF にする必要はない)。不採用。
- **後追い補完にも opt-out を設ける**: 分解と挙動を揃える一貫性はあるが、誤補完は読み飛ばせて実務コストが低く、opt-out を設ける便益が薄い。設定 UI が無駄に増える。不採用。
- **競合解決を last-write-wins / 3-way merge で実装する**: 実装コストが高く、ユーザーが書いた値が AI に消える事故リスクがある。「未補完のみ書く」方が単純で安全。不採用。

## Notes

- 本 ADR は ADR-0061 / ADR-0064 が M-β に委譲した「補完タイミング」「競合解決」を確定し、ADR-0066 (schema の項目) と合わせて Issue #244 を完結させる。
- ADR-0017 (分解の非同期自動発火) / ADR-0064 (後追い補完の auto / 入力要件) はいずれも本 ADR に覆されない。本 ADR は ADR-0017 に opt-out 制御を足し、ADR-0064 が委譲した競合解決を確定する位置づけ。
- 実装は別 issue で扱う: 分解の per-task トグル + グローバル設定 + 手動分解ボタン (新規 roadmap issue が必要)、後追い補完 (#245)。
- ADR-0065 の割り込みボタンと手動分解ボタンのアイコン整理 (割り込みは「緊急 / 注意」を想起させるアイコンに寄せ、分解ボタンと視覚的に区別する) は UI 実装詳細であり ADR 化しない。実装 issue 側で扱う。
- 将来見直す条件: dogfooding で分解性能が安定し、誤分解の削除コストが無視できる水準になったら、per-task トグル / グローバル設定の必要性を再評価する。
