# ADR 0025: 補正後見積もりは大小コントラストで併記、取消線・矢印・ラベルは使わない

- **Status**: Accepted
- **Date**: 2026-04-29
- **Related**: `docs/design/vision.md` §「人間の見積もりは信用できない」 / `docs/design/architecture.md` §1.5 / [ADR 0016](./0016-stack-view-decomposition-children-only.md) / [ADR 0023](./0023-estimation-correction-by-category-median.md) / Issue #93

## Context

ADR 0023 / 0024 で「補正倍率を中央値で算出し、Supabase view + TS 純粋関数で計算する」ところまで決めた。残るは **補正後の値をどう見せるか** の判断。

vision の核 (`docs/design/vision.md` §「人間の見積もりは信用できない」) と architecture.md §1.5 が体験設計の制約を強く決めている:

- 「あなたの見積もりは信用できません」ではなく **「45min 確保しています」とさりげなく** 見せる
- **ユーザーの入力を尊重する**（生値は変えない、否定しない）
- 「補正済みです」と **明言しない**
- 補正が的確であれば信頼が積み上がる

この前提下で、Stack View カード / 詳細パネルで「補正後と元値の関係をどう表現するか」の選択肢が複数ある（取消線・矢印・括弧・大小コントラスト・hover・ラベル付き・補正後のみ）。それぞれが vision §1.5 との相性が違うので、明示的に選ぶ判断が必要。

特に「取消線」と「矢印」は技術的には実装が容易で直感的だが、vision §1.5 の体験設計に照らすと **不適切なのに採用されやすい** 表現なので、不採用判断を ADR に残す価値が高い（実装者が後から「取消線でいいじゃん」と戻すのを防ぐ）。

## Decision

### Stack View（Top カード上ゾーン / 行カード Row 1）

1. **補正後を主表示**: 標準サイズ・主色で `45min` を表示。
2. **元値を副表示**: 小さいフォント・muted color（淡い灰色）で `30min` を補正後の隣 or 下に配置。
3. **ラベル無し**: 「元」「補正後」「確保」等のラベルは付けない。
4. **区切り記号は最小限**: 区切るなら middle dot `·` か無区切り。
5. **取消線 / 矢印 / 括弧表記は使わない**。

### 詳細パネル

1. **補正後を主表示**（Stack View と同じ）。
2. **元値は自然文で添える**: 例「あなたの見積もり: 30min」。
3. **倍率・サンプル数の表現は補正の存在を明言しない**: 「同じ種類のタスクは平均 1.5 倍かかっています」のような文脈的な文言で示す。「補正係数」「補正済み」等の言い回しは UI に出さない。

### 補正されない場合（`task_category = null` or サンプル < 閾値）

- 元値（生 `estimated_min`）のみを表示。併記しない。
- 「補正なし」「データ不足」等のラベルも出さない（補正の存在を明言しない原則を守る）。

具体的なフォントサイズ・色トークン・余白の値はパラメータ扱い（実装 issue で確定）で、本 ADR の supersede ではない。

## Consequences

### 肯定的影響

- **vision §1.5 と整合**。「さりげなく」「ユーザー入力を尊重」「補正を明言しない」の 3 原則すべてに合致する。
- **実装の指針が明確**。「ラベル無し」「取消線・矢印禁止」を ADR で固定することで、PR レビュー時の判断ブレが減る。
- **a11y との衝突が少ない**（ADR 0016 / `kozutsumi-frontend-a11y` skill の方針）。テキストとしては 2 つの数値が並ぶだけなので、screen reader / e2e の semantic locator も素直に書ける。
- **詳細パネルとカードで透明度を出し分けられる**。カードは「さりげなく」、詳細パネルは「確認したい時にちゃんと分かる」、という階層が明示される。

### 否定的影響・トレードオフ

- **muted color の元値が見えにくいケースがある**。低コントラスト環境（屋外 / 低品質ディスプレイ）で元値が読めない可能性。アクセシビリティ最低限のコントラスト比は確保する（実装 issue で確認）。
- **「補正されているか / されていないか」が一見区別しにくい**。元値併記の有無で判別はできるが、ユーザーが意識的に見ないと気づかない。これは vision §1.5 の「さりげなく」の意図的な帰結なので許容する。
- **大小コントラストが Stack View の情報密度を圧迫しうる**。ADR 0016 で行カードは 3 行構成。Row 1 に 2 つの数字が並ぶと混雑する可能性。実装で配置を詰める。
- **詳細パネルの自然文は翻訳・ライティングコストが乗る**。「45min 確保しています」「同じ種類のタスクは平均 1.5 倍かかっています」等の文言は実装側で詰める。

## Alternatives considered

- **取消線**: `~~30min~~ 45min`。実装は CSS 一行で済む。だが「あなたの値は間違いでした」と読める表現で、vision §1.5 の **「ユーザーの入力を尊重する」** と直接矛盾する。打ち消しは強い否定の意味を持つので、補正の文脈には不適切。**不採用**。
- **矢印**: `30min → 45min`。補正の方向感が直感的に伝わる。だが「変化を見せる」こと自体が「補正しました」を視覚化することになり、vision §1.5 の **「補正済みと明言しない」** と矛盾する。「変化を強調しない」ことが体験設計上の要件。**不採用**。
- **括弧 + 「元」ラベル**: `45min (元 30min)`。実装は単純だが「元」というラベルが補正の存在を明示する。**安直すぎる**（議論内のユーザー指摘）。**不採用**。
- **Hover / tooltip で元値を表示**: 普段は `45min` だけ。詳細パネルでは合うが、Stack View カードはモバイル / タッチが前提なので hover が無く不適。**Stack View では不採用**、詳細パネルでも自然文で出す方が透明性が高いので採用しない。
- **ラベル付き並列**: `見積 30min / 確保 45min`。役割が明示されるが文字密度が上がり、ADR 0016 の Stack View 行カード 3 行構成を圧迫する。詳細パネルには合うが、Stack View では不採用。詳細パネルでは「自然文で添える」案を採ったので、ラベル付き並列も実質不採用。
- **補正後のみ**: 元値を出さず `45min` だけ。最も「さりげない」が、ユーザー入力への透明性がゼロで「自分の入力が無視されている」という不信感を生むリスク。vision §1.5 の「ユーザーの入力を尊重する」は「内部で生値を保持する」だけでなく **UI でも生値が確認できる** ことを含む解釈。**不採用**。

## Notes

- 元値（生 `estimated_min`）と補正後（`estimated_min × 倍率`）の **関係を user に説明する文言** は実装 issue (#93) で詰める。本 ADR は「ラベルを付けない」「取消線・矢印を使わない」までを決め、自然文の具体は code 側で更新可能にしておく。
- 補正の倍率が **1 を跨ぐ category（例: コーディング系 0.8x）** で「短く確保される」表示になる時の体験は、本 ADR の方針で対応できる想定（補正後 24min / 元値 30min を併記）。実装後に「短く確保されると user が不安に感じる」現象が観測されたら別 ADR で見直す。
- 本 ADR を supersede する trigger:
  - 補正の存在を user に **明示的に伝える** 方針に変える（vision §1.5 の解釈変更）
  - 補正後の値を **削除して** 提示するモード（user 設定 / power user 機能）を入れる
  - Stack View / 詳細パネル以外の新しい表示面（通知 / メール / 週報等）で本方針が成立しなくなる
- 詳細パネルの倍率・サンプル数の文言は、Phase 4 の行動パターン分析 (architecture.md §1.6) と連動する可能性がある（「ドキュメント系は午後にばらつきが大きい」等）。Phase 4 で別 ADR が立つ可能性あり。
