# ADR 0007: 同期トリガーは手動ボタン + 起動時遅延実行から始める

- **Status**: Accepted
- **Date**: 2026-04-22
- **Related**: [ADR 0005](./0005-google-calendar-sync-via-route-handler.md)

## Context

Google Calendar 同期をいつ走らせるか。候補:

1. **手動トリガー**: ユーザーが「同期」ボタンを押した時のみ
2. **起動時実行**: アプリを開いた時に毎回実行
3. **起動時遅延実行**: 起動時かつ前回同期から閾値（例: 15 分）経過時のみ
4. **定期実行 (cron)**: 一定間隔で自動実行（Vercel Cron / Supabase scheduled functions）

Phase 2 は仮説検証フェーズで、追加インフラを増やしたくない。
ユーザーが想定する利用パターン（必要な時にアプリを開く個人ツール）では、
起動時に最新状態が取れていれば実用上十分。

## Decision

以下の組み合わせで開始する:

1. **手動ボタン**: UI 上に明示的な「カレンダーを同期」ボタンを配置
2. **起動時遅延実行**: アプリ起動時、前回同期から 15 分以上経過していれば
   バックグラウンドで自動実行（non-blocking）

定期実行 (cron) は Phase 2 の範囲外。

## Consequences

### 肯定的影響

- **追加インフラ不要**。Vercel Cron も Supabase scheduled functions も立ち上げない。
- **起動体感を保てる**。non-blocking なので画面表示は遅延しない。
- **無駄な API 呼び出しを抑制**。閾値で短時間の連続起動では走らない。
- **手動ボタンの存在で「同期されてない不安」を解消できる**。ユーザーが明示的に
  最新化を強制できる。

### 否定的影響・トレードオフ

- **アプリを開かないと同期されない**。バックグラウンドで Google Calendar から
  受け取ることはない。リマインダー的な使い方には向かない（Phase 2 の範囲外）。
- **複数デバイスで開くと閾値判定が独立して走る**。各デバイスで一度ずつ走る可能性が
  あるが、idempotent（ADR 0006）なので問題なし。

## Alternatives considered

- **起動時に毎回同期**: 起動体感が悪化、API 浪費。閾値付き遅延の方が良い。不採用。
- **手動のみ**: ユーザーが押し忘れるとデータが古いまま。起動時遅延と組み合わせる方が UX 良。不採用。
- **cron 定期実行**: Phase 2 では追加インフラ。手動 + 起動時遅延で実用上十分なら
  恒久的に不要の可能性もある。Phase 2 範囲外。

## Notes

- 閾値（15 分）は code の constant として実装（ADR 化しない）。
- 最終同期時刻の永続化は ADR 0006 の `syncToken` と同じ仕組みに乗せる想定（実装 issue で確定）。
