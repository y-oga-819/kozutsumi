# ADR 0067: AI 分解・補完の発火はユーザートリガーを基本とし、自動発火は opt-in にする

- **Status**: Accepted
- **Date**: 2026-05-16
- **Related**: `docs/design/vision.md` / [Issue #243](https://github.com/y-oga-819/kozutsumi/issues/243) / [Issue #244](https://github.com/y-oga-819/kozutsumi/issues/244) / [Issue #245](https://github.com/y-oga-819/kozutsumi/issues/245) / [ADR-0017](./0017-ai-task-decomposition-async.md) / [ADR-0058](./0058-timer-three-verbs-and-no-ai-interruption.md) / [ADR-0061](./0061-ai-decomposition-one-hour-target-and-done-condition-schema.md) / [ADR-0064](./0064-task-creation-title-only-with-ai-template-fill.md) / [ADR-0066](./0066-decompose-completion-criteria-deliverable-done-first-step.md)

## Context

ADR-0061 と ADR-0064 はいずれも Notes で「補完タイミング / ユーザー手動入力との競合解決」を M-β 設計時に確定すると後続へ委譲した。本 ADR は Issue #244 として、AI 分解 / 補完の **発火モデル** と **競合解決** を確定する (schema の項目は ADR-0066 で確定済み)。

現状と問題:

- ADR-0017 Decision 2 で AI 分解はタスク作成時に fire-and-forget で **自動発火** する。
- ADR-0064 は AI 後追い補完の存在を決め、Alternatives で「default オフ (ユーザーが明示的に呼ぶ)」を不採用とした。不採用理由は「『AI が裏で動く』前提が vision の受動的体験 (「気づいたら細かくなってる」) と整合する」。
- AI 後追い補完 (#245) は既存タスク — ユーザーが手動編集済みかもしれない — に AI が完了条件を書き込む経路で、ユーザー編集と AI 補完の更新競合が起きうる。ADR-0064 はこの競合解決を未確定のまま残していた。

競合を「解決」するより「構造的に起こさない」方が単純で確実だと判断した。AI が書くのはユーザーが明示的にトリガーした時だけにすれば、ユーザー編集と AI 書き込みが同時に走らない。

## Decision

AI 分解 (decompose) と AI 後追い補完 (#245) の発火を以下に確定する:

1. **発火はユーザートリガーを基本とする。**
   - 分解: スタックのタスクカード上のボタンを押すと分解が走る。分解時に完了条件 (ADR-0066) も同時生成される。
   - 後追い補完: タスク詳細画面のボタンを押すと、未補完の完了条件を AI が埋める。
2. **自動分解は opt-in。** タスク作成フォームにトグルを置き、ON ならタスク登録後に分解が自動発火する。**トグルのデフォルトは OFF。** ユーザー設定でこのデフォルトを ON に変更できる。
3. **競合解決は「競合させない」設計で代替する。** AI が完了条件を書くのはユーザーがトリガーした時だけなので、ユーザー編集と AI 書き込みは同時に起きない。加えて AI 実行中 (`decompose_status='decomposing'`) は当該タスクの完了条件編集を UI でロックする。

ADR-0017 / ADR-0064 との関係:

- 本 ADR は ADR-0017 Decision 2 の「タスク作成時に自動発火」を「opt-in トグル ON 時のみ自動発火」に変更する。ADR-0017 の非同期 (fire-and-forget / core path を AI から独立させる) という核は変更しない — ボタン or トグルで発火した分解は引き続き非同期で走る。
- 本 ADR は ADR-0064 Alternatives が不採用とした「default オフ」を採用する。ADR-0064 の入力要件 (title 必須 / メモ書き推奨 / 補完 schema は ADR-0061 と一致) は変更しない。

## Consequences

### 肯定的影響

- 競合解決ロジックを実装せずに済む。「ユーザーがトリガーした時だけ AI が書く」+「AI 実行中ロック」で構造的に競合が起きない。
- ユーザーが分解の発生タイミングを完全に制御できる。分解したくないタスク / Gemini 呼び出しを増やしたくない状況で勝手に走らない。
- ADR-0058「AI 介入は timer 文脈外でのみ発動」と整合する。AI 介入が能動操作になり、ユーザーから見て予測可能になる。
- opt-in トグル + 設定により、受動的体験を望むユーザーは従来の自動発火に戻せる (vision 受動性の退路を残す)。

### 否定的影響・トレードオフ

- vision「気づいたら細かくなってる」の受動的体験がデフォルトでなくなる。ボタンを押す能動操作になり、受動性を望むなら設定で取り戻す必要がある。これは ADR-0017 / ADR-0064 が前提にした受動性からの転換であり、本 ADR で意識的に受け入れる。
- ユーザーが分解ボタンを押し忘れると、でかいタスクが分解されないまま stack に残り「ゴール駆動の分解」(ADR-0057) が発動しない。トグル / 設定で緩和するが、onboarding で発火経路を伝える必要がある。
- タスクカードにボタンが増える (ADR-0065 の割り込みボタンと同居)。アイコン / 配置の整理が実装で必要になる。

## Alternatives considered

- **全自動のまま、AI 実行中の編集ロックだけ追加する**: 競合はロックで部分的に防げるが、ロック外のタイミング (補完完了直後にユーザーが編集 → 次の補完で上書き) が残る。発火をユーザー制御にする方が単純。不採用。
- **作成時分解は自動を維持し、後追い補完だけ手動にする (分割案)**: 競合回避の観点では十分 (新規タスクは競合しない)。しかしユーザーが「全タスクが勝手に分解される」「Gemini 呼び出しが作成のたびに走る」を制御できない。発火モデルを 2 経路で非対称にすると onboarding と設定 UI も複雑になる。一貫して「ユーザートリガー基本 + opt-in 自動」に倒す。不採用。
- **競合解決を last-write-wins / 3-way merge で実装する**: 実装コストが高く、ユーザーが書いた値が AI に消される事故リスクがある。「競合させない」方が安全。不採用。

## Notes

- 本 ADR は ADR-0061 / ADR-0064 が M-β に委譲した「補完タイミング」「競合解決」を確定し、ADR-0066 (schema の項目) と合わせて Issue #244 を完結させる。
- 実装は別 issue で扱う: 分解ボタン + 作成フォームの自動分解トグル + デフォルト設定 (新規 roadmap issue が必要)、後追い補完ボタン (#245)。現状の作成時自動発火 (ADR-0017) を opt-in トグル経由に変える改修も含む。
- ADR-0065 の割り込みボタンと分解ボタンのアイコン整理 (割り込みは「緊急 / 注意」を想起させるアイコンに寄せ、分解ボタンと視覚的に区別する) は UI 実装詳細であり ADR 化しない。実装 issue 側で扱う。
- 将来見直す条件: dogfooding で「分解ボタンを押し忘れて大タスクが放置される」が頻発したら、デフォルト ON 化や発火リマインドを再検討する。
